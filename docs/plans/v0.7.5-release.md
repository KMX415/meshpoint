## MeshCore companion set-name

### Problem

Users have asked to rename the USB companion from the Meshpoint dashboard. Today:

- **Device name** (`device.device_name`) and **Meshtastic identity** (`transmit.long_name` / `short_name`) are editable under Configuration â†’ Identity.
- **MeshCore companion name** is read from the companion firmware via `MeshCoreTxClient.get_radio_info()` â†’ `self_info["name"]`. **Send Advert** rebroadcasts that name; there is no save path.

### Upstream API (already in `meshcore>=2.1.0`)

The Python library exposes companion protocol command **CMD_SET_ADVERT_NAME (0x08)** as:

```python
result = await meshcore.commands.set_name(name: str)  # -> Event OK or ERROR
```

`set_name` returns **OK/ERROR only**; it does **not** emit `SELF_INFO`. After a successful set, call `send_appstart()` so the client's `self_info` cache updates (same pattern as channel sync uses `get_channel` / `set_channel`).

**Validation:** Strip whitespace, reject empty, cap at **32 UTF-8 bytes** (conservative; firmware limits vary with location/unicode per MeshCore docs). Surface companion `ERROR` payload in API 400 responses.

### Meshpoint implementation

**Backend**

| Piece | Change |
|-------|--------|
| `src/transmit/meshcore_tx_client.py` | `async def set_companion_name(self, name: str) -> SendResult`: `set_name` â†’ on OK `send_appstart()` â†’ `_run_post_command()`. |
| `src/api/routes/config_routes.py` (or small dedicated route module) | `PUT /api/config/meshcore/companion-name` body `{ "name": "..." }`. Requires companion **connected** (503 otherwise). Admin-only if v0.7.3+ auth middleware applies to config PUTs. |
| `MeshcoreConfig` + `local.yaml` (optional polished scope) | `companion_name: Optional[str]` â€” desired name re-applied on USB connect (mirror `sync_channels` hook after connect). |

| Piece | Change |

| `config_enrichment` / `GET /api/config` | No schema change if name still comes from live `companion_name` after refresh. |

**Tests**

## MQTT Home Assistant discovery: state topics (P1)

### Problem

With **Home Assistant discovery** enabled, Meshpoint publishes retained configs under `homeassistant/sensor/meshpoint_{node_id}/.../config` (correct). Those configs declare:

- `state_topic`: `meshpoint/{node_id}/telemetry` (battery, temperature)
- `json_attributes_topic`: `meshpoint/{node_id}/position` (device tracker)

**Nothing is published on `meshpoint/#` today.** Live MQTT traffic uses the Meshtastic layout only:

- Protobuf: `{topic_root}/{region}/2/e/{channel}/{gateway_id}` (e.g. `msh/US/2/e/LongFast/!gateway`)
- JSON mirror (when `publish_json: true`): `{topic_prefix}/2/json/{channel}/{gateway_id}`

Payload shape on `/json/` is a wrapper (`from`, `type`, nested `payload`) â€” not the flat `value_json.battery_level` HA templates expect. Result: HA entities auto-create but stay **unknown** because no messages arrive on the declared `state_topic`.

`HomeAssistantDiscovery.announce_node()` in `src/relay/mqtt_publisher.py` only fires discovery when a published packetâ€™s `decoded_payload` already contains `battery_level`, `temperature`, or `latitude`; it never publishes state to the topics it advertises.

### Proposed v0.7.5 scope (pick one approach)

**Recommended: dedicated `meshpoint/` state topics (matches existing discovery strings)**

| Piece | Change |
|-------|--------|
| `HomeAssistantDiscovery` | On TELEMETRY / POSITION (and relevant NODEINFO fields), publish compact JSON to `meshpoint/{node_id}/telemetry` and `meshpoint/{node_id}/position` (retain or QoS 1 per HA norms). Flat keys: `battery_level`, `temperature`, `latitude`, `longitude`, `altitude`. |
| Discovery configs | Keep `state_topic` / `json_attributes_topic` as today; templates stay `{{ value_json.battery_level }}` etc. |
| `mqtt_publisher.publish()` | Call HA state publish after main Meshtastic publish (same two-gate allowlist). |

**Alternative (not recommended for v0.7.5):** Point discovery at `msh/.../2/json/...` and rewrite `value_template` to dig into the Meshtastic JSON envelope. Fragile across channels, multiple gateways on one broker, and Meshtastic map consumers.

### Acceptance

1. Enable `publish_json` + `homeassistant_discovery`; broker receives discovery on `homeassistant/...`.
2. After a node sends TELEMETRY on an allowed channel, subscribe to `meshpoint/{node_id}/telemetry` and see JSON with `battery_level` / `temperature`.
3. HA sensor entities leave `unknown` and show live values without manual MQTT sensor YAML.
4. POSITION updates populate `meshpoint/{node_id}/position` for the device tracker.

### Tests

- Unit test: TELEMETRY publish triggers `meshpoint/{id}/telemetry` payload with expected keys.
- Unit test: discovery config `state_topic` matches actual publish topic.

### Docs

- `docs/MQTT-AND-MESHRADAR.md` Home Assistant section: clarify that HA sensors use `meshpoint/{node_id}/...` state topics (separate from Meshtastic `msh/...` map traffic).

### Effort

~0.5 day.

### References

- `src/relay/mqtt_publisher.py` â€” `HomeAssistantDiscovery` (`state_topic` lines ~283â€“311)
- `src/relay/mqtt_formatter.py` â€” `MeshtasticMqttFormatter.format_json()` (`/2/json/` topics)

## MeshCore channels: slot alignment, hashtag, send (P1)

### Problem (Discord report, v0.7.4)

Users configuring MeshCore companion channels in **Configuration â†’ MeshCore** see:

1. **Hashtag / public-style channels:** Only private channels (name + 32-char hex key) save successfully. Rows with blank, `null`, or missing keys are **silently skipped** (`update_meshcore_channels` requires both `name` and `key_hex`). MeshCore hashtag channels normally use the `#channelname` label with an **empty or all-zero** 16-byte secret, not a PSK.
2. **8-channel cap:** UI (`_MC_MAX_CHANNELS = 8`) and `sync_channels()` (`_MAX_SLOTS = 8`) stop at 8 rows. MeshCore firmware supports more (~40). Extras are ignored with a log warning.
3. **Off-by-one routing (confirmed in code):** The dashboard shows slot **0 = Public** (locked) and user channels as slots **1â€“7**, but `sync_channels()` writes `channel_keys` starting at **device slot 0**. Messages list uses `enumerate(..., start=1)` for names while TX/RX use raw `channel_idx` from the companion. Symptom: traffic on device slot 0 appears under the first **named** channel in Messages; Public and private channels look **swapped/shifted**. Send from a named channel may target the wrong slot.

### Proposed v0.7.5 scope

| Piece | Change |
|-------|--------|
| `sync_channels()` | Reserve device slot **0** for Public (do not overwrite with first configured key). Write `channel_keys` to slots **1â€¦N** to match Messages UI and `GET /api/messages/channels`. |
| `update_meshcore_channels` | Allow **hashtag** entries: name required, `key_hex` optional; treat empty/missing key as 16 zero bytes (`00` Ă— 32 hex chars). Document: include `#` in the name if your mesh uses it (`#mychannel`), no YAML quotes needed in the dashboard field. |
| `meshcore_card.js` | Clarify Public row hint; optional â€śHashtag (no key)â€ť row template; raise cap toward firmware max or document why 8 (product call). |
| `docs/CONFIGURATION.md` / `COMMON-ERRORS.md` | MeshCore channel table: Public vs private vs hashtag, save + **Save Channels** + confirm in native MeshCore app. |
| Tests | Extend `tests/test_meshcore_channels.py`: slot-0 public preserved; first user channel on slot 1; hashtag with zero key; round-trip channel index in Messages. |

### Acceptance

1. Configure `#test` with zero key: saves, syncs to companion slot 1+, shows in Messages on the correct channel.
2. RX on companion Public (slot 0) appears under **Public** in Messages, not under the first private channel.
3. TX from Messages channel N uses companion slot N.

### Effort

~1 day (sync + API + UI + docs + tests).

## gpsd location source (P1)

v0.7.4 Configuration â†’ GPS shipped **placement only** (static
lat/lon/alt, UART vs static toggle). v0.7.5 lands a real live-GPS
path through the `gpsd` daemon, plus a richer dashboard so

### Domain model

| `StaticSource` (wizard lat/lon/alt) | `src/hal/location/static_source.py` |

### Coordinator wiring

`src/coordinator.py`:

- `start()` instantiates the configured `LocationSource` via the
  factory, awaits `source.start()`, kicks off
  `_location_refresh_loop()` (a periodic task that calls
  `_apply_latest_location_fix()` every
  `update_interval_seconds`).
- `_apply_latest_location_fix()` reads `GpsStatus` from the
  source; if `mode >= min_fix_quality` and coordinates differ
  from `_config.device.{lat,lon,alt}`, mutates the device
  config in place. Downstream consumers (NodeInfo broadcaster,
  MQTT publisher, upstream client, dashboard) read from
  `device.*` and pick up live fixes without further plumbing.
- `stop()` cancels the refresh task and awaits `source.stop()`
  for clean shutdown.

## Local node grid: sort + filter (P2)

### Problem (1kOhm, April 2026 â€” long-standing backlog)

`node_cards.js` today supports only one navigation primitive: a free-text search box (`#node-search`) that filters on long-name, short-name, and node_id. The constructor sets `this._sortBy = 'last_heard'` but the value is **never actually consulted** by `_render()`; observed sort order is the side-effect of `updateFromPacket` calling `this._nodes.unshift(...)` whenever a packet arrives, which places most-recently-heard at the top by accident, not by design.

For operators tracking dozens to thousands of nodes, this is too thin:

- No way to sort by signal strength when triaging weak links.
- No way to sort by hop count when looking for direct vs relayed cohorts.
- No way to sort alphabetically when looking up a specific named node.
- No way to filter the grid down to direct-only or relayed-only nodes.
- The node panel is a fixed width; on wide monitors the cards are narrower than they need to be, on narrow monitors they crowd the map.

All of the data is already in the `nodes` table and on the in-memory node objects (`hop_count`, `latest_rssi`, `latest_snr`, `last_heard`, `long_name`).

| Option | Sort key |

| Name | `(long_name \|\| short_name \|\| node_id)` ascending, locale-aware case-insensitive |

Selection persists in `localStorage` under `meshpoint.nodeCards.sortBy` so reload preserves operator intent. Default `last_heard` matches today's de-facto behavior, so existing users see no surprise.

| All (default) | no filter |

## Favorite / pin nodes (P2)

### Problem (Discord, iceice400)

> With Meshpoint tracking hundreds (or thousands) of localized Meshtastic and MeshCore nodes simultaneously, the Node Cards list and Live Map can get incredibly crowded. It is currently difficult to quickly track your own personal nodes, nodes owned by close friends, or important local repeaters/gateways without searching for them manually every time.

The existing search box (`node_cards.js`) requires retyping after every page reload. There is no way to mark a node as "important" once and have the dashboard remember it.

### Storage decision: localStorage for v0.7.5 MVP, SQLite later if demand surfaces

iceice400 proposed three options. Recommend **localStorage** (his option B) for v0.7.5 for these reasons:

| Question | localStorage | SQLite column |
|---|---|---|
| Consistent with existing patterns? | Yes (`meshpoint.displayUnits`, `meshpoint.mapView`) | Pattern would be new for browser-side preferences. |
| Cross-device sync? | No (per-browser) | Yes (PC + phone see the same list). |
| Multi-operator handling? | Different operators on a household / workshop Meshpoint can have different favorites without stepping on each other. | One shared list; first operator wins. |
| Schema migration? | None. | New `is_favorite INTEGER DEFAULT 0` column on `nodes` table; idempotent `ALTER TABLE` on first boot, mirroring the `role` migration in `database.py`. |
| Backend API + tests? | None. | New `POST /api/nodes/{node_id}/favorite` + `DELETE` + route tests. |
| Survives node-table prune? | Yes (favorites are keyed by node_id; the node row coming back later just rejoins the favorited set). | If we ever add edge-side TTL pruning (we don't today; cloud has 30-day TTL), favorites would orphan unless held in a separate table. |
| Wall time | ~0.5 day | ~1 day plus migration testing. |

The "PC + phone" cross-device case is real but secondary on the **local** dashboard (operators usually access from one main browser inside their network). If users complain after v0.7.5 ships, the path forward is to add a backend favorites table and have the frontend read localStorage first, then migrate up to backend on first boot post-update. localStorage MVP is forward-compatible.
